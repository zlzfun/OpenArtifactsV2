# OpenArtifacts 发布

把一个自包含的 HTML 或 Markdown 文件发布为托管页面,用户在浏览器打开,发布新版本时页面原地更新。

## 发布流程

1. 把页面内容写到项目内或临时目录的一个 `.html` 或 `.md` 文件。
2. 运行发布脚本:

```bash
python3 {SKILL_DIR}/scripts/publish.py <文件路径> --title "简洁的标题" --favicon "📊" [--label "本次改动几个字"]
```

3. 把脚本输出的 URL 展示给用户。

**更新语义(重要)**:同一文件路径重复发布 = 同一 URL 的新版本;不同文件路径 = 全新 artifact。要更新已发布的页面,编辑原文件后用同一路径再发一次。favicon 在同一 artifact 的多次发布间保持不变,用户靠它找到浏览器标签页。

服务地址默认 `http://127.0.0.1:8787`,可用环境变量 `OPEN_ARTIFACTS_SERVER` 覆盖。

## 页面约束(CSP 硬限制)

页面在严格 CSP 下渲染,违反的资源会静默加载失败:

- **禁止一切外部请求**:不能引用 CDN 脚本、外部样式表、Google Fonts、远程图片,不能 fetch/XHR/WebSocket。所有 CSS/JS 必须内联,图片用 data: URI 或直接画 SVG。
- **无后端**:纯静态页面,表单数据无处提交,交互状态只活在页面内。
- **单页**:相对链接不解析;多段内容用页内锚点。
- **只写 body 内容**:服务端会包一层 `<!doctype html>` shell(含最小 reset),不要自己写 `<html>/<head>/<body>` 标签,`<style>/<script>` 直接内联在内容里即可。
- 渲染后不超过 16 MiB。

## HTML Slides(放映页)

面向"给同事展示/汇报"的内容,发布为 slides 类型:文件名用 `*.slides.html`(或加 `--type slides`)。同一份内容自动获得两种看法:**放映模式**(左右翻页,默认)和**滚动模式**(连贯长页面),右下角 HUD 可切换。

**运行时已提供、你不要自己写**:翻页(←/→、空格、点击左右边缘、触摸滑动)、页码与进度条、`#/3` 直达某页、双模式切换、全屏、打印(每节一页的讲义 PDF)。你只负责内容和视觉设计。

**授权契约(违反会导致某个模式下排版损坏):**

- 每页一个顶层 `<section class="slide">`,顺序排列;section 自身的定位、尺寸、内外边距由运行时管理,你的样式只作用于 section 内部的元素。
- **所有尺寸用 rem,布局用 flex/grid 流式,不用绝对定位、不用 vw/vh。** 放映模式根字号 24px、画布 1280×720;滚动模式根字号 17px、列宽 52rem——同一套 rem 排版在两种模式下自动成立。
- 整场背景色放在 `body` 上;个别强调页可给 section 自己的背景(滚动模式下呈现为分节色块)。
- 放映场景要**低密度、大字号**:每页一个观点,正文不小于 1rem,要点不超过 5 条;内容在放映模式下垂直居中,超高会出现页内滚动条——那是内容太多的信号,拆页。
- **像素画布逃生舱**:确需精确摆放的页(手绘架构图等)用 `<section class="slide canvas">`,内容必须是**整页一个 `<svg viewBox="0 0 1280 720">`**,内部一律 px 坐标、不用 rem,不混放 HTML 元素。此类页在放映时是整页画布,滚动模式下作为等比缩放的 16:9 图块嵌入文档流,占比要少。

## 设计要求

以小型设计工作室主创的姿态对待每个页面:根据任务本身校准处理方式,做有意图的配色、字体和版式选择,拒绝模板化。

**先读需求定调。** 文档、计划、备忘录用克制的实用风——仍要有真正的排版层级、考究的间距和成套的配色,但不要过度设计,不需要巨大的 hero。落地页、工具、要分享的作品才值得编辑化的强设计。拿不准时:一个构图良好的页面永远不会错,过度设计有时会。

**写代码前先写设计计划**,再严格照着执行:
- 配色:4–6 个命名的 hex 色值;
- 字体:至少两个角色——克制使用的展示字体 + 互补的正文字体(数据/说明可加第三个);
- 版式:一两句话的布局概念。

**基本功(每个页面都适用):**

- **优先复用项目的设计系统**:先查 CLAUDE.md 或主题/token 文件。优先级永远是:用户的原话 > 项目已有系统 > 你自己的选择。
- **从主题本身取材**:页面的独特感来自内容所属的世界(它的材料、工具、行话)。全程用真实内容,不用占位文本。
- **中性色也要"被选择"**:纯中灰显得没经过思考;向主色偏一点色相的灰才像选出来的。
- **双主题用 token 实现**:调色板定义为 `:root` 上的 CSS 自定义属性,`@media (prefers-color-scheme: dark)` 里只重定义 token,组件永远只消费 token,不在媒体查询里直接写组件样式。暗色主题要同等用心,不是简单反色。
- **排版**:正文行宽约 65 字符;定好 type scale 就不偏离;标题加 `text-wrap: balance`;大写标签加一点 letter-spacing;数字对齐的地方用 `font-variant-numeric: tabular-nums`。CSP 禁止字体 CDN,要么用系统字体栈,要么把字体以 data URI 内联进 `@font-face`。
- **间距交给布局**:同级元素用 flex/grid + `gap`,不用会塌陷的逐元素 margin;宽内容(表格、代码、图)放进自己的 `overflow-x: auto` 容器,页面本体不许横向滚动。
- **文案是设计材料**:从用户视角命名(人管理的是"通知",不是"webhook 配置");控件写清楚会发生什么;错误信息说明原因和解法。
- **结构即信息**:编号、分隔线、标签必须编码内容的真实属性,不是装饰——内容真的有顺序才用 01/02/03。
- 图表和示意图优先用 SVG 或 HTML/CSS 手绘,不要内嵌位图;大数据集做汇总呈现,不要全量内联。
- 纯文档类内容(报告、清单、说明)直接用 `.md` 发布,服务端会渲染成带样式的页面。

**仪表盘/工具类页面**:被扫视和操作,不是从头读到尾。摘要先于细节;状态用形状编码(pill、chip、色条),让需要注意的东西一眼可见;语义色(正常/警告/严重)独立于主色,不算你的 accent;可交互的东西要看起来可交互;给键盘焦点可见状态,尊重 `prefers-reduced-motion`。

**避免"AI 味"设计**:暖米色底 (#F4F1EA) + 衬线展示字 + 赤陶色 accent;近黑底 + 单个荧光绿/朱红点缀;紫蓝渐变 hero;Inter/Space Grotesk 当保底字体;emoji 当章节标记;所有东西都居中;到处 `rounded-lg` 圆角卡片。用户明确要求这些风格时照做,否则别把自由度花在这些默认值上。

## 典型用法

- **走读改动**:渲染 diff 并在边栏加注释,按严重程度着色。
- **方案对比**:多个布局/方案摆在一页网格里,每个下面一行取舍说明。
- **交互调参**:滑块/开关绑定参数,实时预览效果;可加"复制为 prompt"按钮把结果带回会话。
- **进度跟踪**:长任务把清单页面持续重发,用户开着链接就能看到进展(查看器会自动刷新到最新版)。
